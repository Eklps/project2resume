# Quality Check Prompt: 简历质量检查清单

## 目的

在生成简历内容后，执行以下质量检查，确保输出达到面试级别的简历标准。

---

## 1. 内容完整性检查

### project 模式

- [ ] **项目名称**：包含项目名称 + 角色定位
- [ ] **时间范围**：有时间占位符（如无法推断）
- [ ] **技术栈标签**：完整列出主要技术，按类别分组
- [ ] **项目描述**：2-3 句话，包含领域 + 产品形态 + 核心价值
- [ ] **核心职责**：3-5 条，动词开头
- [ ] **技术亮点**：2-4 条，PSR 格式完整

### full 模式（额外检查）

- [ ] **个人信息占位符**：姓名、联系方式等占位符存在
- [ ] **求职意向**：有岗位名称占位
- [ ] **教育背景占位符**：学校、专业、时间
- [ ] **专业技能**：根据扫描结果自动生成
- [ ] **项目经历**：同 project 模式检查

---

## 2. PSR 结构检查

对每条技术亮点执行：

- [ ] **P (Problem) 存在**：明确描述了技术挑战/业务场景
  - ❌ 反例：直接写方案，没说为什么
  - ✅ 正例："在秒杀场景下面临高并发库存超卖风险"

- [ ] **S (Solution) 存在**：包含具体技术方案
  - ❌ 反例："优化了性能"
  - ✅ 正例："采用 Redis + Lua 脚本实现原子性库存预扣减"

- [ ] **R (Result) 存在**：有效果描述，最好可量化
  - ❌ 反例："提升了性能"
  - ✅ 正例："QPS 从 XX 提升至 XX [需验证]，超卖率降至 0"

- [ ] **标签正确**：[高并发] / [性能优化] / [架构设计] 等标签与内容匹配

---

## 3. 事实准确性检查

- [ ] **技术栈验证**：所有技术栈在依赖文件中有据可查
  ```
  检查方法：
  - 对照 package.json / pom.xml 中的依赖
  - 确认版本号一致
  - 未使用的技术不应出现
  ```

- [ ] **量化指标可追溯**：
  ```
  检查方法：
  - API 数量是否通过 grep 统计得出
  - 代码行数是否通过 wc -l 统计得出
  - 推断的性能数据是否标注了 [需验证]
  ```

- [ ] **架构描述准确**：
  ```
  检查方法：
  - 分层描述是否和实际目录结构一致
  - 模块名称是否和代码中一致
  ```

---

## 4. 语言质量检查

### 动词力度

- [ ] **无 C 级弱动词**：搜索以下弱动词并替换
  ```
  弱动词清单（应替换）：
  - "编写" → "设计" / "实现"
  - "完成" → "交付" / "构建"
  - "参与" → "主导" / "负责"
  - "做了" → "设计并实现"
  - "写了" → "开发" / "实现"
  - "用了" → "引入" / "采用"
  ```

### 术语规范

- [ ] **大小写统一**：
  ```
  正确：Spring Boot, MyBatis-Plus, Redis, RabbitMQ, MySQL, Docker, Nginx
  错误：springboot, mybatis plus, redis, rabbitmq, mysql, docker, nginx
  ```

- [ ] **版本号格式**：使用 `X.x` 格式（如 `Spring Boot 3.x`）

### 中英文混排

- [ ] **中英文之间有空格**：`使用 Redis 实现缓存` ✅
- [ ] **英文与标点之间无空格**：`Redis，用于缓存` ✅
- [ ] **数字与中文之间有空格**：`实现 45 个 API` ✅

---

## 5. 反模式检查

- [ ] **无功能罗列**：
  ```
  检查方法：
  搜索"实现了...、...、..."的模式
  如果一条描述包含 3 个以上并列功能且无技术深度 → 不合格
  ```

- [ ] **无技术堆砌**：
  ```
  检查方法：
  技术栈标签行之外，正文中不应出现纯技术列表
  ```

- [ ] **无空泛描述**：
  ```
  检查方法：
  搜索"优化了"、"提升了"、"改进了"但后面没有具体方案和数字的
  ```

- [ ] **无职责不清**：
  ```
  检查方法：
  搜索"参与了"、"协助了"等模糊词
  ```

---

## 6. 岗位匹配检查

- [ ] **目标岗位关键词覆盖**：
  ```
  后端岗：应包含至少 3 个 → 架构/并发/缓存/数据库/事务/API设计
  前端岗：应包含至少 3 个 → 组件化/状态管理/性能优化/响应式/跨端
  全栈岗：前后端各至少 2 个关键词
  ```

- [ ] **技术亮点与岗位相关**：
  ```
  后端岗的技术亮点不应全是前端内容（反之亦然）
  ```

---

## 质量评分

```
质量分 = (通过项数 / 总检查项数) × 100

等级:
- 90-100: 优秀 ⭐⭐⭐⭐⭐  可直接投递
- 80-89:  良好 ⭐⭐⭐⭐     微调后可投递
- 70-79:  合格 ⭐⭐⭐       需要补充和完善
- 60-69:  需改进 ⭐⭐       建议重新优化
- <60:    不合格 ⭐         需要大幅修改
```

## 修复建议格式

对于未通过的检查项，提供修复建议：

```
❌ 问题: [描述]
📍 位置: [章节/条目]
💡 建议: [具体修复方案]
📌 优先级: 高/中/低
```

## 输出格式

```
📊 简历质量检查报告

总分: [分数]/100
等级: [等级]

✅ 通过: [数量] 项
❌ 未通过: [数量] 项

[如有未通过项]
需要修复:
1. ❌ [问题] — [位置] — [建议] — [优先级]
2. ...

💡 整体建议:
- [建议1]
- [建议2]
```
