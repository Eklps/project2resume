# 简历写作方法论

## 核心理念

> 简历不是项目文档，是**面试提纲**。每一条描述都应该能引出一个有深度的面试问题。

---

## 1. PSR 法（Problem - Solution - Result）

### 定义

PSR 是技术简历中描述项目亮点的黄金框架：

| 要素 | 含义 | 简历中的体现 |
|------|------|-------------|
| **P** (Problem) | 面临的技术挑战 | "在...场景下，面临..." |
| **S** (Solution) | 采用的技术方案 | "采用...方案，通过..." |
| **R** (Result) | 达到的效果 | "实现...，提升/降低...%" |

### 示例对比

**❌ 无 PSR（功能罗列）**：
> 使用 Redis 做了缓存，使用 RabbitMQ 做了异步处理

**✅ 有 PSR（深度展示）**：
> 针对秒杀场景下的库存超卖风险(P)，采用 Redis + Lua 脚本实现原子性库存预扣减，
> 结合 RabbitMQ 异步下单解耦流量(S)，秒杀接口 QPS 提升至 XX，超卖率降至 0(R)

### PSR 写作技巧

1. **P 要具体**：不要说"性能差"，要说"在百人并发下响应时间超过 3s"
2. **S 要有层次**：不只写用了什么，要写怎么用的
3. **R 要量化**：能用数字就用数字，不能用数字就用定性描述

## 2. STAR 法（Situation - Task - Action - Result）

### 定义

STAR 法更适合描述完整的项目经历或解决一个复合问题的过程：

| 要素 | 含义 | 适用场景 |
|------|------|----------|
| **S** (Situation) | 项目背景 | 项目描述部分 |
| **T** (Task) | 你的任务/角色 | 核心职责部分 |
| **A** (Action) | 具体行动 | 技术亮点部分 |
| **R** (Result) | 最终结果 | 量化成果 |

### PSR vs STAR

| 维度 | PSR | STAR |
|------|-----|------|
| 粒度 | 单条技术亮点 | 整个项目/任务 |
| 侧重 | 技术深度 | 完整叙事 |
| 适用 | 简历的技术亮点 | 面试的行为问题 |
| 推荐 | 简历优先使用 | 面试回答使用 |

## 3. 动词选择策略

### 动词力度金字塔

```
         ┌──────────┐
         │  架构     │  ← 从 0 到 1，顶层设计
         │  主导     │
         ├──────────┤
         │  设计     │  ← 方案级别
         │  构建     │
         ├──────────┤
         │  优化     │  ← 改进已有系统
         │  重构     │
         │  实现     │
         ├──────────┤
         │  开发     │  ← 日常工作（较弱）
         │  集成     │
         │  部署     │
         ├──────────┤
         │  编写  ❌ │  ← 避免使用
         │  参与  ❌ │
         │  完成  ❌ │
         └──────────┘
```

### 动词替换对照表

| 弱动词 | 强动词替换 | 示例 |
|--------|-----------|------|
| 编写了 | 设计并实现 | 设计并实现了分布式锁方案 |
| 做了 | 构建 / 搭建 | 构建了完整的 CI/CD 流水线 |
| 用了 | 引入 / 采用 | 引入布隆过滤器解决缓存穿透 |
| 参与了 | 主导 / 负责 | 主导后端服务架构设计 |
| 完成了 | 交付 / 落地 | 交付高可用的秒杀系统 |
| 写了 | 开发 / 实现 | 实现基于角色的权限控制 |
| 解决了 | 攻克 / 排查并修复 | 攻克分布式场景下的数据一致性问题 |
| 改了 | 重构 / 优化 | 重构核心业务模块，代码可读性提升 |

## 4. 量化指标策略

### 可从代码推断的指标

| 来源 | 可量化内容 | 简历表述 |
|------|-----------|----------|
| grep API 注解 | API 数量 | "设计并实现 45 个 RESTful API" |
| grep 实体/表定义 | 数据表 | "设计 12 张核心数据表" |
| wc -l | 代码量 | "核心模块代码量 1.2 万行" |
| 测试文件统计 | 测试覆盖 | "编写 23 个单元测试用例" |
| 模块目录 | 模块数 | "划分 5 大业务模块" |

### 需要用户确认的指标（标注 [需验证]）

| 类型 | 推断方法 | 需用户确认 |
|------|----------|-----------|
| QPS/TPS | 从架构模式推断 | 实际压测数据 |
| RT 降幅 | 从缓存/索引优化推断 | 实际前后对比 |
| 可用性 | 从高可用方案推断 | SLA 数据 |
| 用户量 | 从 README 或部署规模推断 | 实际运营数据 |

### 量化表述模板

```
性能类: "接口 RT 从 XXms 降低至 XXms [需验证]，降幅 60%"
并发类: "支撑 XX 级并发请求 [需验证]，超卖率降至 0"
规模类: "服务 XX 名用户 [需验证]，日均处理 XX 笔订单"
效率类: "问题定位效率提升 70% [需验证]"
质量类: "单测覆盖率达 XX%"
```

## 5. 项目描述公式

### 一句话描述公式

```
{领域} 领域的 {产品形态}，{核心功能描述}，服务 {用户群体}
```

**示例**：
- "面向高校用户的二手商品交易平台，支持商品发布、搜索、即时通讯和在线交易"
- "企业级任务调度系统，支持分布式 Cron 任务编排和实时监控"
- "移动端社交电商应用，集成直播带货、社群营销和智能推荐"

### 职责描述公式

```
{强动词} + {技术方案/架构} + {量化范围}
```

**示例**：
- "设计并实现基于 Spring Boot 的微服务架构，划分用户、商品、订单、消息 4 大服务模块"
- "架构 Redis 多级缓存体系，涵盖本地缓存 + 分布式缓存 + 缓存预热三层策略"
- "主导数据库表结构设计，规范化至第三范式，包含 12 张核心业务表"

## 6. 技术亮点分类指南

根据代码中发现的技术模式，选择最匹配的亮点标签：

| 标签 | 适用场景 | 代码线索 |
|------|----------|----------|
| [高并发] | 秒杀、抢购、限流 | Redis+Lua, Redisson, Sentinel |
| [性能优化] | 缓存、索引、查询优化 | Redis, 布隆过滤器, SQL优化 |
| [架构设计] | 分层、模块化、设计模式 | AOP, 工厂模式, 策略模式 |
| [安全] | 认证、授权、加密 | JWT, Spring Security, OAuth |
| [高可用] | 熔断、降级、重试 | Sentinel, Hystrix, 重试机制 |
| [消息驱动] | 异步处理、事件驱动 | RabbitMQ, Kafka, 事件总线 |
| [数据一致性] | 分布式事务、幂等 | Seata, 分布式锁, 幂等设计 |
| [DevOps] | CI/CD、容器化、监控 | Docker, GitHub Actions, Prometheus |
| [搜索] | 全文搜索、智能推荐 | Elasticsearch, 倒排索引 |
| [实时通信] | 即时消息、推送 | WebSocket, SSE, STOMP |

## 7. 常见反模式与修正

### 反模式 1: 功能罗列

```
❌ "实现了用户登录、注册、修改密码、忘记密码、第三方登录功能"
✅ "设计基于 JWT + Spring Security 的统一认证体系，支持 Token 无感刷新
    和多端登录互踢，接入微信 OAuth2.0 第三方登录"
```

### 反模式 2: 技术堆砌

```
❌ "使用了 Spring Boot, MyBatis-Plus, Redis, RabbitMQ, MySQL, Docker, Nginx, Vue"
✅ 在技术栈标签行统一展示，正文中只在 PSR 亮点中提及具体技术
```

### 反模式 3: 空泛描述

```
❌ "优化了接口性能"
✅ "针对商品列表接口 RT 过高问题，引入 Redis 缓存 + MySQL 联合索引优化，
    接口 RT 从 800ms 降至 120ms [需验证]"
```

### 反模式 4: 过度承诺

```
❌ "系统支撑百万级并发"（学生项目实际没这个量级）
✅ "通过压测验证系统可支撑 XX 级并发 [需验证]"
   诚实标注，面试时可以讲方案而非吹数字
```
